## 1. 现状与挑战 (Current Status & Challenges)

### 1.1 The [Metric]% Wall ([Metric]% 召回率之墙)
在 Phase 1 的 MVP 验证中，我们基于 **"[Methodology Name, e.g., Task-Document Matrix]"** 对 [Product Name] 现有文档进行了切片与索引测试。
*   **测试结果:** 在 [N] 个真实用户场景的“对抗性测试”中，仅有 **[X] 个** 能够召回正确文档，**召回率 (Recall Rate) 为 [Metric]%**。
*   **失败根因:**
    *   **小白搜不到专家话:** 用户搜 "[User Term]", 文档写 `[Technical Term]`.
    *   **现象搜不到原理:** 用户搜 "[Symptom]", 文档写 `[Root Cause/Action]`.
*   **结论:** [Metric]% 的召回率意味着 **[100-Metric]% 的用户请求将直接转为工单**。这在生产环境中是不可接受的，它直接导致了高昂的服务成本和糟糕的用户体验。

### 1.2 The "Zero Headcount" Reality (零人力现实)
面对这 [100-Metric]% 的缺口，传统的解决方案是“加人写文档”。
*   **现实约束:** 我们没有额外的人力资源 (0 Headcount) 来丰富或重构现有的产品文档。
*   **必然选择:** 我们不能依靠“人肉填坑”。我们必须采用 **"AI Augmentation" (AI 增强)** 方案，用算法算力来弥补文档人力的缺失。

### 1.3 Theoretical Basis: Agentic RAG (理论依据)
本方案的技术架构深度参考了 **Agentic RAG** 范式。
*   **核心理念:** 面对企业级海量异构文档，传统的 "Chunks + Vector" 检索模式已失效。我们需要将每一个文档或产品线视为一个独立的 **Tool Agent**。
*   **我们做的:** "Ghost Writer" 生成的虚拟问题和 "Task Matrix" 生成的元数据，本质上是在为这些 Tool Agents 编写 **"Tool Descriptions" (工具说明书)**。
*   **目的:** 只有具备了精准的描述，上层的 **Top Agent (Router)** 才能在用户提问时，精准地调用正确的产品 Agent。

### 1.5 Agent Roles & Responsibilities (角色分工表)
为确保 "Agentic RAG" 架构的高效运转，各层级 Agent 需严格遵守以下分工定义：

| 角色 (Role) | 定义 (Definition) | 触发时机 (Trigger Point) | 运作逻辑 (Operation Logic) |
| :--- | :--- | :--- | :--- |
| **Top Agent**<br>(Router) | **总调度官**。<br>掌握所有产品线的能力边界，但不涉及具体细节。 | 用户输入 Query 时<br>(Entry Point) | 1. **意图识别**: 判断用户是在问“A”、“B”还是“C”。<br>2. **任务分发**: 将模糊需求转化为特定 Tool 的调用指令 (e.g., `call Agent_A`). |
| **Product Agent**<br>(Tool Agent) | **领域专家**。<br>如 `[Product A] Agent`。<br>只懂自己的一亩三分地，但懂得很深。 | 被 Top Agent 调用时<br>(Sub-task Execution) | 1. **检索**: 在本产品的 Vector/Graph 索引中查找相关切片。<br>2. **生成**: 基于检索到的切片生成最终答案。<br>3. **回传**: 将结果返回给 Top Agent 汇总。 |
| **RAG Engine**<br>(Infrastructure) | **知识仓库**。<br>包含向量索引、摘要索引和知识图谱。 | Product Agent 需要“查资料”时 | 提供多维度的召回能力（语义相似度 + 结构化关联）。 |

## 2. 核心解决方案 (Core Solution)
| 痛点 | 现象 | 解决方案 | 预期效果 |
| :--- | :--- | :--- | :--- |

## 3. 演进路线图 (Roadmap)

### Phase 1: MVP 单点突破 (Current Status)
*   **目标:** 跑通 [Single Product] 单产品的智能索引流程，验证技术可行性。
*   **交付物:**
    *   `task_matrix.md`: 任务-文档映射标准。
    *   `generate_tags.py`: 自动化打标脚本。
    *   `full_index.json`: 包含元数据的 Sidecar Index。
*   **状态:** ✅ 完成 (基准召回率 [Metric]%)。

### Phase 2: Search Quality Tuning (搜索质量调优)
*   **目标:** 在“零人力写文档”的前提下，通过 LLM 增强索引，解决“小白问题搜不到专家文档”的顽疾。
*   **行动计划 (Project Ghost Writer):**
    1.  **逆向问题生成 (Reverse Question Generation):**
        *   遍历文档，让 LLM 生成对应的“大白话”问题。
    2.  **查询重写 (Query Expansion):**
        *   引入 RAG Rewrite 层，将用户 Query 翻译为专业术语。
    3.  **对抗性测试:**
        *   扩充 `test_queries.json`，覆盖更多 Edge Case。
        *   目标召回率 > [Target]%。

### Phase 3: Platform Scale (全平台覆盖)
*   **目标:** 将 MVP 经验复制到全平台 [N]+ 产品，建立平台级文档治理标准。
*   **行动计划:**
    1.  **引入交叉产品:** 接入更多异构文档库。
    2.  **标准化输出:** 发布《文档智能索引接入规范》(SOP)。
    3.  **统一端点:** 提供统一的 Search API，支持多产品混合检索。
